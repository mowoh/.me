Unit 5: Design Patterns III - Behavioural Patterns

This Unit provides an Introduction to Six Key Behavioural Design Patterns:
 Strategy: Allows you to define a family of algorithms, encapsulate each one, and make them interchangeable.
Observer: Defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified.
Chain of Responsibility: Passes a request along a chain of handlers, where each handler either processes the request or passes it to the next handler.
Template Method: Defines the skeleton of an algorithm in a method, deferring some steps to subclasses.
Command: Encapsulates a request as an object, allowing you to parameterise clients with different requests.
State: Allows an object to alter its behaviour when its internal state changes. Required reading: Nesteruk, D. (2022) Design patterns in modern C++20 : reusable approaches for object-oriented software design. Second edition. California: Apress, Pt I, II, III

Objectives

  • Understand and explain what design patterns are and why they are important in software development.
  • Identify and apply key creational design patterns, including Singleton, Factory Method, Builder, Prototype, and Abstract Factory.
  • Recognise how creational patterns are used in real-world software systems to improve flexibility and maintainability.
  • Implement the Factory Method pattern in a practical coding exercise

    • Task

    You will implement the Strategy pattern in a project, giving you hands-on experience with applying behavioural patterns.

    Solution Steps

  • Define a Strategy interface with a method for processing payments (e.g., process_payment)
    • .
  • Create concrete strategies for each payment method (e.g., CreditCardStrategy, PayPalStrategy, CryptoStrategy).
  • Define a Context class that uses a strategy to process payments.
  • Allow the client to dynamically switch between payment strategies at runtime.

    • Solution

    
										
from abc import ABC, abstractmethod

# Strategy Interface
class PaymentStrategy(ABC):
    @abstractmethod
    def process_payment(self, amount):
        pass

# Concrete Strategies
class CreditCardStrategy(PaymentStrategy):
    def process_payment(self, amount):
        return f"Processing ${amount} via Credit Card"

class PayPalStrategy(PaymentStrategy):
    def process_payment(self, amount):
        return f"Processing ${amount} via PayPal"

class CryptoStrategy(PaymentStrategy):
    def process_payment(self, amount):
        return f"Processing ${amount} via Cryptocurrency"

# Context
class PaymentProcessor:
    def __init__(self, strategy: PaymentStrategy):
        self.strategy = strategy

    def set_strategy(self, strategy: PaymentStrategy):
        self.strategy = strategy

    def execute_payment(self, amount):
        return self.strategy.process_payment(amount)

# Client Code
processor = PaymentProcessor(CreditCardStrategy())
print(processor.execute_payment(100))  # Output: Processing $100 via Credit Card

processor.set_strategy(PayPalStrategy())
print(processor.execute_payment(50))   # Output: Processing $50 via PayPal
